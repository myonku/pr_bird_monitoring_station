# bms_copilot 演进设计说明

版本：0.1.0
状态：Draft
适用范围：`bms_copilot`

## 1. 文档目的

本文档用于定义 `bms_copilot` 分区的演进方向、目录边界与阶段目标。

本分区的定位不是“再去调用其他服务节点的一个外壳”，而是一个自包含的 AI 助手服务节点：

- 能力上可以覆盖与其他节点相似的基础智能交互能力。
- 实现上保持轻量，不把核心能力拆到外部服务节点。
- 工具与底层资源交互都在 `bms_copilot` 内部完成。
- 后续若与系统主链路集成，也应通过统一入口层接入，不通过 tool 去跨节点调用其他服务。

## 2. 当前状态判断

当前 `bms_copilot` 已经具备较完整的“分层骨架”，但还不是可稳定运行的 agent 服务。

已存在的主要结构包括：

- `agent_core`：agent 的核心编排、知识、记忆、工具协议、模型 provider 占位。
- `modules`：面向具体业务能力的共用实现目录，适合承载真实 tool 逻辑。
- `repo`：底层数据访问与外部资源适配层。
- `services`：服务级能力封装，如通信、认证控制、引导协调、公共服务。
- `iface`：各层之间的抽象接口。
- `models`：领域模型、配置模型、输入输出契约。
- `app`：服务装配与应用入口层。

从现状看，目录已经铺开，但核心执行链仍然偏空壳，说明当前阶段最重要的任务不是扩展功能数量，而是先把职责边界、启动骨架和最小闭环落到位。

## 3. 分层原则

### 3.1 顶层封装

`app` 和 `services` 负责把 `bms_copilot` 组织成一个完整服务节点。

- `app` 负责启动装配、配置注入、运行时组合。
- `services` 负责对外提供节点级能力封装，例如通信、认证、统一流量入口、生命周期管理。

这两层的职责是“把服务跑起来并对外呈现为一个节点”，不承担具体 agent 推理逻辑。

### 3.2 核心中枢

`agent_core` 是服务节点的智能核心，负责：

- 意图识别
- 任务规划
- 工具编排
- 结果聚合
- 会话级上下文组织
- 模型 provider 抽象

`agent_core` 只保留核心决策与编排逻辑，不直接承载具体业务 tool 的实现细节。

### 3.3 工具实现下沉

所有具体 tool 的实现放在 `modules` 目录下。

这类模块的特点是：

- 面向具体能力，而不是面向编排。
- 允许直接和底层资源交互，例如 `repo`、本地模型、缓存、向量库、文件系统。
- 作为 `bms_copilot` 内部的共用能力复用，而不是挂在 `agent_core` 里。

`agent_core` 通过接口或适配器调用这些模块，而不是反向把具体实现嵌进去。

### 3.4 数据访问边界

`repo` 只负责底层资源适配：

- 数据库
- 向量库
- 缓存
- 文件/本地持久化
- 外部模型或检索所需的基础客户端

`repo` 不承担 agent 规划，不承担业务编排，也不承担 tool 协议。

## 4. 关键设计约束

1. `bms_copilot` 内部 tool 不调用其他服务节点，不把外部节点当作 tool 的常规实现路径。
2. `agent_core` 只做核心编排，不放具体 tool 代码。
3. `modules` 承载可复用的工具实现，并允许直接访问 `repo`。
4. `services` 负责把 agent 能力包装成统一的节点服务能力。
5. 运行期应优先采用本地/内部资源闭环，避免引入跨节点依赖造成复杂化。
6. 工具、provider、记忆、检索、推理应在接口层解耦，便于后续替换实现。
7. 设计上保持单 Agent、显式编排、可观测、可回放。

## 5. 目标能力域

`bms_copilot` 初期应覆盖的能力域如下：

- 自然语言搜索与查询
- 统计问答
- 物种知识补充与检索增强
- 图片识别请求编排
- 会话级上下文管理
- 统一响应生成与解释

这些能力的优先级不是“先做完整聊天”，而是“先把可验证的业务能力接起来”，让 agent 真正成为任务编排器。

## 6. 推荐目录角色

### 6.1 `app`

建议承载：

- 服务启动入口
- 配置加载与注入
- 依赖装配
- 生命周期管理

### 6.2 `services`

建议承载：

- 通信服务封装
- 统一入站/出站处理
- 认证与流量控制的节点级包装
- 节点运行时协调

### 6.3 `agent_core`

建议承载：

- intent / planning / orchestration
- memory / knowledge / evaluation
- provider 与工具调用抽象

### 6.4 `modules`

建议承载：

- search 工具实现
- statistics 工具实现
- inference 工具实现
- knowledge / RAG 工具实现
- session / history 辅助能力

### 6.5 `repo`

建议承载：

- MySQL / Redis / Milvus / MongoDB / Etcd 适配
- 文件与本地缓存访问
- 统一查询与持久化客户端

## 7. 演进阶段

### 阶段 0：骨架收口

目标：把结构从“目录齐全”变成“边界清晰”。

- 明确 `app`、`services`、`agent_core`、`modules`、`repo` 的职责。
- 统一配置加载方式与启动入口。
- 定义 tool、provider、memory、knowledge 的接口面。
- 保证空实现不会误导为可用功能。

验收标准：

- 能说清楚每一层负责什么，不存在职责重叠。
- 核心路径上每个模块的输入输出契约明确。

### 阶段 1：最小闭环

目标：跑通一个最简单的 agent 请求链路。

- 接收 `AgentRequest`。
- 做最基础的意图识别。
- 调用 1 到 2 个本地 tool。
- 生成统一响应。

建议先做的闭环：

- 文本查询 -> 搜索工具 -> 汇总回答。

验收标准：

- 一条请求可以完整穿过 agent_core、modules、repo。
- tool 调用链可记录、可追踪、可测试。

### 阶段 2：核心能力补齐

目标：把鸟类观测场景真正需要的能力补完整。

- 搜索能力。
- 统计能力。
- 物种知识检索能力。
- 图片识别编排能力。
- 会话上下文能力。

验收标准：

- 每类能力都有独立模块。
- 能以统一接口复用到不同请求类型。

### 阶段 3：知识与记忆增强

目标：让回答更稳定、更可解释。

- 引入文档切分与检索。
- 引入向量检索与引用组装。
- 引入会话短期记忆和运行记录。

验收标准：

- 回答可附证据来源。
- 会话内上下文能稳定续接。

### 阶段 4：模型与评估体系

目标：让系统可切换、可比较、可回归。

- provider 抽象统一化。
- 支持多模型切换。
- 建立请求级日志、工具耗时、错误码、评估样本。

验收标准：

- 同一业务能力可替换不同模型源。
- 有基本的质量回放与回归测试能力。

### 阶段 5：服务化收束

目标：把 bms_copilot 从工程骨架收束成稳定服务节点。

- 固化 app / services 装配。
- 固化 tool / repo 依赖边界。
- 固化配置、启动、监控与错误语义。

验收标准：

- 启动、运行、调试、扩展路径稳定。
- 新 tool 能按模块方式接入，不污染 agent_core。

## 8. 近期实施顺序建议

如果按当前进度继续推进，建议优先顺序是：

1. 补齐启动入口与配置装配。
2. 把 agent_core 的接口链补完整，但先不填复杂逻辑。
3. 在 modules 中落第一个真实 tool。
4. 让 tool 通过 repo 访问本地数据或索引。
5. 用一个最小请求链路把编排、工具、响应串起来。

## 9. 非目标

本阶段不建议优先做的事情：

- 把 tool 逻辑塞回 agent_core。
- 让 tool 通过其他服务节点完成常规能力。
- 一开始就做复杂多智能体自治。
- 一开始就追求全量场景覆盖。

## 10. 结论

`bms_copilot` 的正确演进方式不是“先堆模型”，而是先把它做成一个结构清楚、边界稳定、内部自给自足的轻量 agent 服务节点。

在这个前提下，`agent_core` 负责聪明地编排，`modules` 负责具体能力实现，`repo` 负责底层资源接入，`app/services` 负责把它包装成一个真正可运行的服务节点。
