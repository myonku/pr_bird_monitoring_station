# 客户端开发任务清单

状态：草案

## 1. 目标

先完成一个可演示、可切换、可后续接后端的 Flutter 客户端骨架。当前阶段不追求复杂业务闭环，重点是把页面框架、基础状态、mock 数据和模式切换先立住。

## 2. 开发原则

- 先做壳，再做内容。
- 页面先用 mock 数据跑通，后续再替换真实接口。
- 页面结构优先稳定，业务逻辑保持可替换。
- 认证、会话、接口层先预留，不在第一阶段过度实现。

## 3. 第一阶段范围

第一阶段只做下面这些内容：

- 项目基础结构整理。
- 应用入口、路由和底部导航。
- 登录页和主框架页。
- 首页、记录页、统计页、我的页的基础展示。
- 本地 mock 数据和简单状态管理。
- development / no-auth 模式切换的基础能力。

第一阶段不做下面这些内容：

- 复杂认证流程。
- 真正的后端接口联调。
- 复杂表单和高交互业务。
- 过度细化的权限控制。

## 4. 任务拆分

### 4.1 项目骨架

- 确定 `lib/` 下的基础目录结构。
- 统一应用入口、路由和主题入口。
- 建立页面基类和通用组件的放置位置。

### 4.2 界面框架

- 完成登录页。
- 完成底部导航主框架页。
- 完成首页、记录页、统计页、我的页的占位和基础布局。
- 保证页面之间可以正常跳转。

### 4.3 数据层

- 为首页、记录、统计、用户信息准备 mock 数据。
- 通过 repository 或 service 层统一管理数据来源。
- 页面只依赖数据抽象，不直接依赖具体 mock 结构。

### 4.4 状态管理

- 维护当前模式状态。
- 维护登录态占位状态。
- 维护首页、记录、统计、个人信息的基础展示状态。

### 4.5 模式切换

- 支持 development 和 no-auth 两种模式。
- development 模式保留后续 token 流程入口。
- no-auth 模式保持登录入口，但只做业务展示，不保存凭证。

### 4.6 后端预留

- 预留登录、刷新、登出、当前用户信息接口。
- 预留首页概览、记录列表、记录详情、统计数据接口。
- 后续替换真实接口时尽量不改页面层。

## 5. 推荐推进顺序

1. 先整理项目目录和入口。
2. 再完成路由与底部导航框架。
3. 然后把登录页和四个主页面占位搭起来。
4. 接着补 mock 数据与基础状态管理。
5. 最后补模式切换和后端接口预留层。

## 6. 阶段验收标准

第一阶段完成后，至少满足以下条件：

- 应用可以正常启动。
- 登录页可以进入主界面。
- 四个主页面可以切换和展示。
- 页面数据全部来自 mock 层。
- 可以切换 development / no-auth 的基础模式。
- 代码结构足够清晰，后续可以直接接真实接口。

## 7. 后续迭代方向

- 第一轮：补齐页面内容和基础交互。
- 第二轮：接入真实后端接口。
- 第三轮：补认证、刷新和退出逻辑。
- 第四轮：再考虑更细的体验优化和功能扩展。
