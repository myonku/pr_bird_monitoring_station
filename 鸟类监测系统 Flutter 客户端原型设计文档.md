# 鸟类监测系统 Flutter 客户端原型设计文档

## 1. 文档目的

本文档用于指导鸟类监测系统 Flutter 客户端的原型开发，作为进入实现阶段的直接参照。客户端主要承担**用户登录、监测数据查看、统计可视化、个人信息管理**等功能，并支持两种运行模式：

- **development 模式**：启用完整认证链路，后续接入真实后端服务
- **no-auth 模式**：用于测试与联调，客户端不保存凭证，业务请求不携带凭证，但仍保留密码登录入口与完整业务界面

此外，在后端尚未完整可用之前，客户端所有接口数据优先使用**本地 mock data**，待后端服务搭建完成后再逐步切换到真实接口。

---

## 2. 项目背景与定位

系统整体为鸟类监测站，包含边缘端采集、推理与上传，后端负责认证与业务数据处理，Flutter 客户端面向用户提供移动端查看能力。

客户端定位为：

> **鸟类监测系统移动端查看与测试客户端**

主要目标：

1. 提供登录入口
2. 展示监测站运行概况
3. 浏览鸟类识别记录
4. 查看统计分析结果
5. 在 no-auth 模式下支持纯业务测试
6. 在后端未完成前，使用 mock data 完成前端开发与演示

---

## 3. 开发原则

### 3.1 原型优先

当前阶段不追求完整复杂业务，优先保证：

- 页面结构完整
- 认证流程清晰
- 数据展示闭环可运行
- 后端可平滑替换

### 3.2 以 mock 驱动开发

初期所有页面接口统一使用本地 mock data：

- 登录返回 mock token 或 mock 登录状态
- 首页统计数据使用 mock 概览数据
- 记录列表使用 mock 记录集
- 图表数据使用 mock 统计集
- 用户信息使用 mock profile

待后端服务搭建完成后，仅替换数据源层，不改页面层结构。

### 3.3 模式可切换

客户端需支持两种模式：

- **development**
- **no-auth**

切换方式可通过：

- 环境变量
- 启动配置
- 本地配置文件
- Debug 面板

建议开发阶段直接用 `dart-define` 或本地配置控制。

## 4. 模式说明

### 4.1 development 模式

适用于后续正式联调。

特点：

- 启用登录认证
- 保存 access token / refresh token
- 接口请求自动携带凭证
- token 失效后自动刷新
- 登录失败或刷新失败跳转登录页

### 4.2 no-auth 模式

适用于测试联调与功能验证。

特点：

- 客户端仍保留**用户名/密码登录入口**
- 登录成功后**后端不返回凭证**
- 客户端**不保存 token**
- 后续业务请求**不携带凭证**
- 除认证链路外，其他业务界面与 development 模式保持一致
- 仅保留纯业务展示和测试能力

说明：

- no-auth 模式下，客户端的“登录”本质上是为了进入测试业务流，而不是建立真实认证会话
- 后端在该模式下也不启用认证校验

## 7. 页面原型说明

## 7.1 登录页

### 页面目标

完成用户身份进入系统的入口。

### 页面元素

- 用户名输入框
- 密码输入框
- 登录按钮
- 模式提示（development / no-auth）
- 可选：记住登录状态

### 行为说明

- 输入用户名和密码后点击登录
- development 模式下：
  - 调用登录接口
  - 成功后保存 token
- no-auth 模式下：
  - 仍执行登录动作
  - 但不接收/不保存 token
  - 登录成功后直接进入主界面

### 开发阶段数据来源

- 本地 mock 登录结果

## 8. 认证与会话设计

## 8.1 development 模式认证流程

1. 用户输入用户名和密码
2. 客户端请求网关登录接口
3. 网关转发认证中心校验
4. 成功后返回 access token + refresh token
5. 客户端保存凭证
6. 后续请求携带凭证
7. 401/403 时尝试刷新 token
8. 刷新失败则回到登录页

## 8.2 no-auth 模式认证流程

1. 用户输入用户名和密码
2. 客户端请求登录接口
3. 后端不启用认证校验
4. 登录成功但不返回凭证
5. 客户端不保存 token
6. 后续业务请求不携带凭证
7. 保持纯业务浏览与测试能力

## 9. 数据策略

## 9.1 初期数据策略

所有页面统一使用本地 mock data，包含：

- 登录结果
- 用户信息
- 首页概览
- 记录列表
- 记录详情
- 统计图表数据

### mock data 要求

- 结构固定
- 命名统一
- 便于后续替换为真实后端接口
- 尽量贴近真实业务字段

## 9.2 后端接入策略

当后端准备完成后，按以下步骤替换：

1. 保留页面层不动
2. 先替换网络请求层
3. 再替换 mock repository
4. 最后接入真实 token 与刷新逻辑

## 14. 接口预留建议

虽然当前全部使用 mock data，但建议从一开始就按真实接口形态封装 repository/service 层，避免后续改动页面。

建议预留接口类型：

### 认证

- 登录
- 刷新
- 登出
- 当前用户信息

### 业务

- 首页概览
- 记录列表
- 记录详情
- 统计数据
- 修改密码

---

## 15. no-auth 模式下的特殊约束

为了符合测试要求，no-auth 模式需满足：

1. 仍保留用户名/密码登录入口
2. 登录后不返回凭证
3. 客户端不保存 token
4. 后续接口请求不带 token
5. 页面结构和业务展示与 development 模式一致
6. 仅作为测试和联调用途，不作为正式认证方案

---

## 16. 开发验收标准

客户端原型完成后，至少满足以下条件：

- 能切换 development / no-auth 模式
- 能完成登录动作
- 能进入主界面
- 能浏览首页、记录页、统计页、个人中心
- 能使用 mock data 完成全部页面展示
- development 模式下具备完整 token 生命周期处理
- no-auth 模式下不依赖认证凭证即可完成业务浏览

---

## 17. 总结

本原型设计的核心策略是：

- **先用 mock data 跑通前端结构**
- **先把页面和状态流搭起来**
- **认证链路按仓库设计分为 development / no-auth 两套行为**
- **no-auth 模式保留登录入口，但不保存凭证**
- **后端完成后只替换数据层，不推倒页面重做**
