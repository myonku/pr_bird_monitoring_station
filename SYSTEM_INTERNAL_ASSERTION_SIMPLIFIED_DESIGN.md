# 系统内部校验简化设计说明（网关签名 + 模块本地验签）

版本：1.0.0  
状态：Draft（待评审）

## 1. 设计背景

现有方向已明确：

- 认证信息由认证中心统一管理与暴露。
- 网关承担外部入口流量处理。
- 非认证中心模块不本地维护认证状态。

在星形流量拓扑下（外部 -> 网关 -> 目标模块，普通模块之间无横向调用），运行期每跳再回认证中心校验会引入额外复杂度与延迟。

本设计目标是将“内部转发校验”从在线回源改为离线断言验证。

## 2. 目标与非目标

目标：

- 网关统一执行主认证与主限流。
- 网关向目标模块转发时附带签名断言。
- 目标模块使用本地公钥目录进行验签与轻量约束校验。
- 保持 commsec 通道加密及复用能力，并要求内部通信在可用安全通道上执行（预热复用或首跳握手）。

非目标：

- 替代认证中心签发权威。
- 放宽网络访问边界控制。
- 引入普通模块之间横向调用机制。

## 3. 核心思路

## 3.1 在线校验 -> 离线断言

旧模式（运行期）：

1. 网关收到请求。  
2. 网关/目标模块调用认证中心校验。  
3. 目标模块再次做认证相关判断。

新模式（运行期）：

1. 网关完成认证中心校验并作限流。  
2. 网关签发内部断言并转发。  
3. 目标模块本地验签、校验约束、执行轻量限流。  
4. 不再回认证中心做该跳校验。

## 3.2 断言字段（建议最小集）

必选字段：

- `ver`：断言版本。
- `alg`：签名算法（建议 `ed25519`）。
- `kid`：网关签名 key id。
- `iss`：签发者（gateway entity id）。
- `aud`：目标服务标识。
- `jti`：断言唯一 ID（重放防护键）。
- `iat` / `exp`：签发与过期时间（短 TTL，建议 5-10s）。
- `principal_id`、`entity_type`、`entity_id`。
- `session_id`、`token_id`（可选但建议保留）。
- `scopes`、`gateway_id`、`trace_id`、`request_id`。
- `method`、`path`、`query_hash`、`body_sha256`。

可选字段：

- `secure_channel_id`（若需绑定到既有通道语义）。

## 3.3 签名与验签

签名：

- 网关使用本地私钥（仅引用，不泄露原文）对规范化 payload 签名。
- 推荐头：`x-internal-assertion`，承载 `header.payload.signature`（类 JWS 紧凑串）。

验签：

- 目标模块基于 `kid`（或 `iss`）从本地 SecretKeyService 查公钥。
- 使用统一 CryptoUtils 按 `alg` 验签。
- 验签成功后校验语义字段约束。

## 3.4 语义校验规则

目标模块必须校验：

- `aud == 本服务标识`。
- `exp` 未过期，`iat` 在时钟容忍范围内。
- `method/path/body_sha256` 与实际请求一致。
- `jti` 未重复使用（Redis TTL 防重放）。
- `kid` 对应公钥状态有效（active）。

若任一失败，直接拒绝（401/403 或协议映射错误码）。

## 4. 安全边界

该简化设计成立的前提：

- 服务发现“注册顺序约束”仍保留（bootstrap/readiness 后注册）。
- 网络层只允许网关访问内部模块入口。
- 公钥目录可读且 key 轮换机制完善。

注意：

- “可注册到服务发现”是运行流程约束，不应单独作为运行期信任凭据。
- 必须保留断言验签与重放防护，不能仅凭头字段明文信任。

## 5. 与现有能力对齐

现有可复用能力：

- SecretKeyService 已支持按 key_id/entity_id 查公钥目录。
- CryptoUtils 已支持 `SignByAlgorithm/VerifyByAlgorithm`。
- commsec 通道 `EnsureChannel` 已支持复用优先。

需新增能力：

- 内部断言模型与签发/验签服务。
- 断言拦截器接入（目标模块入站）。
- `jti` 重放缓存与 TTL 策略。
- 头字段命名统一与兼容过渡。

## 6. 兼容策略（双栈过渡）

说明：本节为历史过渡记录，当前基线已完成过渡并切换到单路径策略。

当前基线：

1. 网关转发仅下发规范化下游头与内部断言。  
2. 目标模块按“本地验签 + 重放防护”执行校验，不再回退旧头明文信任逻辑。  
3. 运行期认证中心回源校验路径已下线（仅保留签发与目录权威职责）。

## 7. 对限流模型的影响

- 主限流：在网关执行（认证后）。
- 次限流：目标模块保留资源保护型限流，不再依赖认证中心回源。
- 限流描述符主体来源：从“原始 header”迁移为“验签后的身份上下文”。

## 9. 风险评估

- 风险：网关私钥泄露导致断言可伪造。  
缓解：私钥引用托管、最小权限、快速轮换、kid 黑名单。

- 风险：时钟漂移导致误拒绝。  
缓解：时钟容忍窗口 + NTP 对齐 + 告警。

- 风险：重复请求被重放。  
缓解：jti + exp + Redis TTL 防重放。

- 风险：字段命名不一致导致身份丢失。  
缓解：先统一契约并提供兼容映射。

## 10. 结论

该简化方案可行，并与当前“认证中心统一管理认证信息”的方向一致。  
推荐采用“网关主校验 + 断言下发 + 模块本地验签 + 强制安全通道”的组合，作为内部转发运行期固定基线。
