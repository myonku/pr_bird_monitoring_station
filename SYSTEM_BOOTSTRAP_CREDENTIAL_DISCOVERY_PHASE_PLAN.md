# 非认证中心模块 Bootstrap-凭证-服务发现闭环设计约束与推进模板

版本：1.0.0
状态：Constraint-First / Template
日期：2026-04-23
适用范围：gateway、data_worker、edge_server、client，以及后续所有需要凭证管理的非认证中心模块

说明：

- 本文件由原“网关 Bootstrap-凭证-服务发现闭环审查与阶段推进清单”重构而来。
- gateway 仍是当前参考实现，但本文件的目标是全局化约束，而不是网关专属阶段表。
- certification_server 不适用本模板：它是权威认证中心，不做同类自管理闭环。

---

## 1. 设计目标

1. 所有需要凭证管理的模块，除 certification_server 以外，都必须具备自动管理自身凭证的能力。
2. 后端模块在 development 模式下，如果参与服务发现注册，则注册、续期和退出必须由凭证状态驱动。
3. 客户端是特殊分支：可以自动刷新会话，但在无法恢复时必须手动重新登录，不进入 re-bootstrap 闭环。
4. 模块的业务处理能力、认证处理能力和凭证生命周期能力必须解耦，不能把凭证管理塞进 AuthControl 或业务控制器内部。

---

## 2. 全局硬约束

1. Bootstrap 的成功定义必须同时满足远端认证成功与本地凭证落地成功；只拿到远端结果但没有可靠持久化，不算成功。
2. 运行期必须持续评估本地凭证状态，不能只在启动阶段检查一次。
3. 凭证仍有效时优先 refresh；refresh 不可用、失败或无法恢复时再执行 re-bootstrap。
4. 凭证失效后，参与服务发现的后端模块必须先退出发现状态，再重新 bootstrap 或重新获取可用凭证后恢复注册。
5. AuthControl 只负责认证决策和流量控制，不负责 bootstrap、refresh、re-bootstrap、注销或本地凭证持久化。
6. 配置只允许在启动期读取一次，运行期只允许使用快照，不允许再从配置源反复回读。
7. no_auth 模式可以短路认证链，但不能留下“表面运行、实际继续使用过期凭证”的灰色状态。
8. 日志和测试必须覆盖 bootstrap、refresh、失效、退出发现、重新恢复这几个状态迁移点。

---

## 3. 模块分类与当前状态

### 3.1 gateway

gateway 是当前参考实现，已经具备自动 bootstrap、refresh、re-bootstrap 和凭证驱动服务发现的完整闭环。

### 3.2 edge_server

edge_server 已经具备自动凭证循环能力：

1. EdgeAuthCoordinator 有 bootstrap、refresh、unauthorized 恢复和 logout 能力。
2. ensure_ready 会先尝试 refresh，失败后再 bootstrap。
3. ensure_startup_ready 在 full_development 模式下执行启动门禁，缺少长期凭证时会触发 bootstrap。
4. no_auth 模式使用占位认证协调器，不执行 bootstrap、refresh 或鉴权恢复。

edge_server 是外部系统，不参与服务发现。

### 3.3 client

客户端已经具备自动刷新会话的能力，但恢复策略和后端不同：

1. 通过本地会话管理器自动判断 access / refresh 是否可用。
2. refresh 失败、会话过期或凭证不完整时，会清理本地会话并要求用户重新登录。
3. 客户端不做 re-bootstrap，也不参与服务发现注册。
4. 这是已经实现的行为，后续文档和实现都应保持这个语义，不要把客户端改写成后端式 bootstrap/re-bootstrap 闭环。

### 3.4 certification_server

certification_server 是权威认证中心，不适用本模板的自管理要求。它不需要像其他模块那样对自身凭证做 bootstrap/refresh/re-bootstrap 闭环，也不应该被写成依赖自身服务发现闭环的模块。

---

## 4. 通用推进模板

下面的阶段顺序适用于后续所有需要凭证管理的非认证中心模块。模块可以按自身协议裁剪细节，但不能跳过核心语义。

### 阶段 A：契约对齐

目标：把模块侧的 bootstrap 或会话契约与认证中心的真实协议对齐。

要求：

1. 明确请求和响应的最小完整字段，不允许只保留阶段标记。
2. 明确哪些字段必须持久化，哪些字段只做临时转发。
3. 明确异常返回语义，确保调用方知道是失败、过期还是需要重新登录。

### 阶段 B：真实凭证接入

目标：把占位凭证、假签名或假握手替换为真实认证材料。

要求：

1. 使用本地密钥或会话材料构造真实请求。
2. 不能依赖固定假值完成 bootstrap 或 refresh。
3. 真正的签名、验签或令牌交换必须在模块内形成可测的闭环。

### 阶段 C：本地凭证快照

目标：让模块能够用本地状态判断“当前凭证是否仍然可用”。

要求：

1. 本地快照至少要能表达主体、会话、令牌、过期时间和凭证状态。
2. 必须能区分可参与后续流程、只可 refresh、必须重建三种状态。
3. 快照写入失败时，bootstrap 或 refresh 不能算成功。

### 阶段 D：状态驱动的参与门禁

目标：把凭证状态和参与能力绑定起来。

要求：

1. 对参与服务发现的后端模块，development 模式下必须由凭证状态决定是否注册、是否续期、是否退出。
2. 凭证无效时，必须停止保持 discoverable 状态。
3. 凭证恢复时，必须重新进入 discoverable 状态。
4. 如果模块本身没有服务发现平面，就不要伪造一个注册流程；等它真正接入服务发现时，再补这一层门禁。

### 阶段 E：运行期自恢复

目标：形成 refresh first、re-bootstrap second 的自恢复链路。

要求：

1. 先尝试 refresh。
2. refresh 成功后，更新本地快照并继续运行。
3. refresh 失败或不再可恢复时，执行 re-bootstrap。
4. 后端模块如果参与服务发现，恢复成功后再恢复注册或续期。
5. 客户端在这一阶段的特殊规则是：refresh 失败后清理会话并要求用户手动重新登录，不做 re-bootstrap。

### 阶段 F：可观测性与测试

目标：让状态迁移可追踪、可回归。

要求：

1. 至少记录 bootstrap、refresh、失效、退出发现、恢复成功和恢复失败这几类阶段日志。
2. 必须有覆盖自动刷新和失效恢复的回归测试。
3. 后端模块如果带服务发现，还要覆盖“凭证失效时退出发现、恢复后重新进入发现”的测试。
4. 客户端要覆盖“刷新失败后清会话并要求重新登录”的测试或等价校验。

---

## 5. 后端模块的通用落地规则

1. gateway、data_worker、edge_server 以及后续所有后端模块，只要需要凭证管理，就必须遵守本模板的 A-F 阶段。
2. 对于实际参与服务发现的后端模块，development 模式下必须额外满足“凭证状态驱动注册”这一约束。
3. 对于没有服务发现平面的后端模块，只需要落地自动凭证循环；不要为了满足模板而人为增加空壳注册逻辑。
4. 如果后续某个模块从“不参与发现”升级为“参与发现”，必须先补齐 D 阶段，再切换发布模式。

---

## 6. edge_server 与 client 的专项说明

### 6.1 edge_server

edge_server 是当前最接近 gateway 的参考对象之一，但它的现状是“有自动凭证循环、无服务发现门禁”。

当前可直接复用的点：

1. 自动 bootstrap。
2. 自动 refresh。
3. 自动 re-bootstrap。
4. 401 / 403 场景下的未授权恢复。
5. no_auth 与 full_development 的模式分流。

当前不能直接假设的点：

1. 没有服务发现注册、注销或 lease keepalive 的现成实现。
2. 不能把“已具备凭证循环”误读成“已经具备凭证驱动注册”。

### 6.2 client

客户端和后端模块不同，它的主体是用户会话，不是可服务发现的后端实例。

当前必须保持的行为：

1. 会话可自动 refresh。
2. refresh 无法恢复时，必须清理本地会话。
3. 用户看到的是“请重新登录”，而不是“重新 bootstrap”。
4. 客户端不参与服务发现，也不需要注册、续期门禁。

---

## 7. 验收口径

1. 对后端模块来说，凭证失效后不能继续把实例留在 discoverable 状态。
2. 对后端模块来说，恢复后必须重新回到可用状态，且恢复过程可追踪。
3. 对 edge_server 来说，当前只要确认自动凭证循环成立即可；若后续加入服务发现，必须先补 D 阶段。
4. 对客户端来说，失败后的正确结果是清会话并要求手动登录，不是重 bootstrap。
5. 对 certification_server 来说，不适用本模板的自管理闭环要求。

---

## 8. 备注

1. gateway 仍然是这套约束的参考实现，不是唯一模板。
2. 本文件后续可继续作为其他非认证中心模块的通用推进清单，只需要把模块名和协议差异替换掉即可。
3. 如果某个模块缺少服务发现平面，就只落地自动凭证管理，不要为了形式完整性硬套注册逻辑。
